## Reconciliation decision

Use the **new prompt as the backbone**. It is stronger because it forces the key strategic correction: **Vesta is not an app with WhatsApp intake; Vesta is a WhatsApp-native household coordinator, with the app as the 20% setup/configuration/review layer.**

My earlier prompt contributes useful pieces, but some of it was still too app-centric. The unified prompt below resolves that.

## Conflicts to resolve

| Conflict            | Earlier prompt                                                              | New prompt                                            | Resolution                                                                                                                  |
| ------------------- | --------------------------------------------------------------------------- | ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |

## Unified Lovable prompt

```text
You are working on Vesta, a WhatsApp-native household logistics assistant for Brazilian families.

Use the existing Vesta Product Spec v9.0 as the source of truth if it is available in the project. If it is not available in the current Lovable context, do not invent unsupported requirements; use this prompt as the implementation source.

Core strategic correction:
Vesta should no longer feel like an app with WhatsApp intake. It should feel like a capable household coordinator that lives in WhatsApp, with the web/mobile app used only for setup, privacy, audit, rules, complex review, calendar connection, memory management, and configuration.

The key product rule:
Approximately 80% of daily user interactions should happen in WhatsApp, and only 20% should happen in the app.

Primary positioning:
“Manda um recado. A Vesta transforma em ação.”

Long-term brand promise:
“A casa aprende.”

Use the learning promise carefully:
- Day 1: “Você define as regras. A Vesta executa.”
- Week 2–4: “Cada aprovação ensina algo.”
- Month 2+: “A casa começou a saber.”

Primary user experience:
A tired caregiver should be able to forward a school message, photo, voice note, boleto, diarista message, condomínio notice, party reminder, medical appointment, or household reminder to Vesta in WhatsApp and get back a concise, actionable response without opening the app.

Do not build a generic chatbot.
Do not build a generic family management dashboard.
Build a household operations assistant.

The core product loop is:
Capture → classify → propose action → ask smallest necessary question → approve → write/delegate/remind → audit → learn only with explicit confirmation.

The desired feeling:
“I forwarded a messy household message, and Vesta handled it.”

Not:
“I signed up for another family management app.”

## 1. Redesign onboarding around WhatsApp-first activation

Replace the current app-heavy onboarding with a progressive WhatsApp-first activation flow.

Activation does not mean “onboarding complete.”
Activation means “first household item approved.”

New onboarding sequence:

### Step 1 — Minimal account creation

Ask only:
- name
- phone number
- city/timezone
- preferred language
- privacy/consent acceptance

Do not ask for full household setup upfront.
Do not ask for every family member upfront.
Do not ask for routines upfront.
Do not ask for calendar connection before the user has seen value.

### Step 2 — Connect WhatsApp immediately

Create a screen called WhatsAppConnectionScreen.

Copy:

“A Vesta trabalha pelo WhatsApp.”

“Encaminhe recados, fotos, áudios e lembretes. A Vesta organiza, pergunta o mínimo necessário e escreve no calendário quando você aprovar.”

Primary CTA:
“Conectar WhatsApp”

Behavior:
- Generate a unique verification token, e.g. VESTA-482.
- Open WhatsApp via deep link with the token prefilled.
- Show verification status.
- App should poll for verification status.
- If real WhatsApp/Twilio integration is not implemented yet, mock verification but structure the code so real backend verification can replace it later.

### Step 3 — First WhatsApp-style interaction

After verification, show a WhatsApp-like chat simulation or continue in real WhatsApp if integration exists.

Opening message:

“Oi, sou a Vesta. Me encaminhe um recado real da casa — escola, consulta, boleto, diarista, condomínio ou festa. Se preferir, mande um áudio.”

The user should process one household item before being asked to complete a full profile.

Example user input:
“Reunião da escola quinta 19h.”

Vesta response:
“Anotei: Reunião da escola — quinta 19h. Vou colocar no calendário e lembrar 1h antes. Quem vai?”

Buttons:
- “Eu”
- “Outra pessoa”
- “Depois”

After approval:
“Feito. Quer conectar seu calendário para eu escrever isso automaticamente?”

### Step 4 — Progressive enrichment

Only after the first useful action, ask contextual setup questions.

Examples:
- “Essa tarefa é para uma criança da casa?”
- “Quer salvar essa escola como contato conhecido?”
- “Tem mais alguém que divide essas tarefas com você?”
- “Quer conectar seu calendário agora para eu escrever os eventos?”
- “Quer convidar outra pessoa da casa?”

Do not interrupt the first-use moment with long setup.

### Track onboarding progress

Prepare or mock these states:

- onboarding_started
- whatsapp_verified
- first_message_processed
- first_item_approved
- calendar_connected
- partner_invited
- setup_completed
- last_step

Show internal progress in development/debug UI if useful, but do not make this feel like a checklist to the user.

## 6. Add app escalation rules

Most approvals should stay in WhatsApp.

Escalate to app only when:

- confidence is low
- there are 4+ linked actions
- payment safety checklist is required
- health data is ambiguous
- external outbound message needs preview
- concierge purchase/payment is required
- memory review has multiple staged items
- privacy/audit view is needed
- calendar conflict needs visual review
- a rule would affect multiple people or recurring actions

When escalating, use specific copy.

Good:
“Esse caso tem 5 ações ligadas. Melhor revisar visualmente.”

Good:
“Esse pagamento precisa de uma checagem antes de salvar.”

Good:
“Tem conflito no calendário. Melhor revisar.”

Bad:
“Open the app to continue.”

Bad:
“Something went wrong.”

Bad:
“Please complete this process in the dashboard.”

Create component:
AppEscalationCard

It should include:
- reason for escalation
- short summary of pending action
- CTA: “Revisar no app”
- secondary CTA where safe: “Agora não”

## 8. Conversation style guide

Vesta’s tone should be Brazilian Portuguese, concise, capable, and warm without being chatty.

Principles:
- Brief by default.
- No corporate language.
- No excessive enthusiasm.
- No generic chatbot phrases.
- No “Como posso ajudar hoje?” as the default.
- Do not pretend to be human.
- Never over-apologize.
- Never moralize household dynamics.
- Always ask the smallest necessary question.
- If uncertain, say so.
- Use household language, not productivity SaaS language.
- Prefer action verbs: anotar, lembrar, confirmar, revisar, salvar, delegar, resolver.

Good:
“Anotei. Pode deixar.”
“Feito.”
“Corrigi.”
“Desfeito.”
“Não tenho isso salvo ainda. Quer me contar?”
“Tem certeza? A consulta está marcada pra quinta, não quarta.”
“Quer que eu salve isso na memória da casa?”
“Esse caso tem 5 ações ligadas. Melhor revisar visualmente.”

Bad:
“Claro! Eu adoraria te ajudar com isso!”
“No momento essa funcionalidade não está disponível.”
“Parece que você está sobrecarregada.”
“Como posso ajudar hoje?”
“Gerencie suas tarefas familiares com inteligência artificial.”
“Open the app to continue.”
```
